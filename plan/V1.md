# Searcher Frontend вЂ” Development Plan

## Overview

A React + TypeScript frontend for searching entries in a self-hosted MeiliSearch instance. The UI is **schema-agnostic**: the current dataset is movies, but swapping to books, songs, or any other index requires editing a single config file.

## Tech Stack

| Layer | Choice |
|---|---|
| Framework | React 19 + TypeScript |
| Build tool | Vite |
| Search UI | `react-instantsearch` + `@meilisearch/instant-meilisearch` |
| Routing | React Router |
| Styling | CSS Modules (Phase 3+) |
| Base CSS | `instantsearch.css/themes/satellite.css` |

## Agnostic Design: `searchConfig.ts`

The entire UI renders from a single configuration object. Swapping datasets means editing only this file.

```typescript
export const searchConfig = {
  indexName: "movies",
  appTitle: "Movie Search",
  titleField: "title",
  imageField: "poster",
  displayFields: [
    { key: "overview", label: "Synopsis", truncate: 150 },
    { key: "genres", label: "Genres", type: "badge" },
    { key: "release_date", label: "Release Date", type: "date" },
  ],
  filterableFields: [
    { key: "genres", label: "Genre" },
  ],
  hitsPerPage: 12,
};
```

## Environment Variables

The frontend reads connection details from environment variables (never committed):

```
VITE_MEILISEARCH_URL=http://localhost:7700
VITE_MEILISEARCH_API_KEY=searchKey
```

These map to the existing `.env` values (`MEILISEARCH_URL` + `MEILISEARCH_API_KEY`), passed through Vite's `VITE_` prefix convention.

## Phases

### Phase 1 вЂ” Bare MVP (Functional, zero styling)

**Goal:** Working search over the configured index with dynamic field rendering.

- Scaffold React + TypeScript app with Vite in `/frontend`
- Install `react-instantsearch`, `@meilisearch/instant-meilisearch`, `instantsearch.css`, `react-router-dom`
- Create `searchConfig.ts` with schema mapping
- Build `<SearchBox>` + `<Hits>` using InstantSearch widgets
- Hit card renders fields dynamically from config (no hardcoded field names)
- Basic detail view: clicking a hit navigates to `/detail/:id` showing all fields from config
- Connect to MeiliSearch via `VITE_` environment variables
- Use `instantsearch.css/themes/satellite.css` defaults вЂ” no custom CSS

**Deliverable:** A working search page with results and a detail page. Looks unstyled but fully functional.

---

### Phase 2 вЂ” Layout & Structure

**Goal:** Organized layout, basic spacing, responsive container.

- Add a header with the app title (from `searchConfig.ts`)
- Wrap search in a centered, max-width container
- Replace infinite scroll with `<Pagination>` widget
- Add "Back to results" link from detail view
- Basic responsive layout: single column mobile, wider desktop

**Deliverable:** Structured, readable layout with no visual polish yet.

---

### Phase 3 вЂ” Visual Polish & Theming

**Goal:** Make it look intentional.

- Implement CSS Modules for component-scoped styles
- Style the search box (larger, centered, prominent)
- Style hit cards (image thumbnail, title, truncated text, genre badges)
- Style detail page (full-width image, structured metadata sections)
- Add configurable theme via CSS variables (accent color, font family, border radius)
- Hover and focus states on interactive elements

**Deliverable:** A visually complete, themed search application.

---

### Phase 4 вЂ” Filtering & Refinement

**Goal:** Let users narrow down results.

- Add `<RefinementList>` for fields defined in `config.filterableFields`
- Add `<SortBy>` if alternate sorting indexes are configured
- Add `<ClearRefinements>` widget
- All filter widgets are optional via config вЂ” no widget renders for unlisted fields

**Deliverable:** Filterable search experience.

---

### Phase 5 вЂ” Robustness & DX

**Goal:** Production-ready quality.

- Loading states and "no results" empty state
- Error boundary for MeiliSearch connection failures
- `<Configure>` widget with sensible defaults from config
- Environment variable validation on app startup
- `README.md` in `/frontend` with setup and configuration instructions

**Deliverable:** Production-ready, documented, resilient search frontend.

---

## Constraints

- **Search-only API key**: Frontend uses `MEILISEARCH_API_KEY` (not the admin key)
- **No hardcoded fields**: All field references come from `searchConfig.ts`
- **Existing infra**: MeiliSearch runs on Docker via the repo's `docker-compose.yaml` on port 7700
- **Node version**: 24.13.0 (per `.nvmrc`)
